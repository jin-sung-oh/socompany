  - 이전 3D 자산은 현재 확정된 6종 동물/7개 역할 체계와도 맞지 않아 새 기준의 오피스 뷰로 보기 어려웠음

  - 3D 뷰의 1차 상호작용은 플레이어 조작 복제가 아니라 `사원 클릭 → 카메라 포커스 → 현재 작업/위치 확인` 흐름으로 제한

  - 현재 제품의 안정된 상호작용은 2D 쪽에 있으므로 3D는 먼저 "상태를 읽는 커맨드 덱" 역할로 두는 편이 구현 리스크가 낮음
  - 기본값까지 3D로 바꾸면 기존 사용 흐름을 갑자기 흔들 수 있어, 선택 가능한 대체 뷰로 두는 쪽이 더 안전함

## [2026-03-11] 3D 오피스 대화는 proximity 복제 대신 선택 기반 직통 채널로 시작
- 배경:
  - 3D 씬 재설계 후에도 실제 메시지 전송/응답 흐름이 없어, 상태 확인은 되지만 상호작용은 2D보다 약한 상태였음
  - 2D의 proximity 대화는 플레이어 이동과 거리 판정 위에 얹혀 있는데, 현재 3D 뷰에는 아직 플레이어 아바타와 공간 조작이 없음
- 결정:
  - 3D에서는 사원을 클릭하면 하단 대화 패널 preview가 열리고, 여기서 바로 `직통 대화`를 시작하는 흐름을 채택
  - 대화 생성에는 기존 `spatialPrompt`와 채팅 저장소를 그대로 재사용해 종/역할/위치/주변 에이전트 문맥은 유지
  - 패널 문구는 2D proximity 전용 표현을 그대로 쓰지 않도록 공용 `OfficeConversationPanel`에 커스텀 텍스트 props를 추가
- 이유:
  - 플레이어 조작이 없는 상태에서 proximity를 억지로 흉내 내기보다, 선택 기반 직통 채널이 현재 3D 뷰와 가장 자연스럽게 맞음
  - 프롬프트와 채팅 저장 구조를 공유해야 2D/3D 전환 시 대화 컨텍스트가 분리되지 않음
  - 공용 패널을 확장해 두면 이후 3D에 플레이어나 근접 채널을 넣더라도 UI를 다시 갈아엎지 않아도 됨

## [2026-03-11] FastAPI 연동은 Fastify 서버가 브리지 역할을 맡는 구조로 시작
- 배경:
  - 사용자는 별도로 만든 FastAPI 서버와 현재 `packages/server` 백엔드가 서로 데이터를 주고받아야 한다고 요청했음
  - FastAPI 서버 코드는 현재 워크스페이스 안에 없어서, 두 서버를 한 번에 직접 패치할 수는 없는 상태였음
- 결정:
  - `packages/server`는 FastAPI의 실제 스펙에 맞춰 `GET /`, `GET /health`, `GET /api/v1/agents`, `POST /api/v1/tasks`, `GET /api/v1/tasks/{id}`를 호출하는 브리지 레이어를 가짐
  - `FASTAPI_BASE_URL`이 설정되면 기존 `enqueueTask`는 로컬 시뮬레이션 대신 FastAPI에 역할별 task를 순차 생성하고, `GET /api/v1/tasks/{id}` polling으로 완료/실패를 추적
  - FastAPI가 나중에 push 방식을 추가할 수 있도록 `/api/integrations/fastapi/events` webhook도 유지하되, 현재 기본 동기화 방식은 polling으로 둠
- 이유:
  - 사용자가 준 FastAPI 스펙은 webhook보다 task 생성과 상태 polling이 중심이므로, 이 흐름에 맞추는 편이 실제 서버와 가장 빨리 맞물림
  - 렌더러는 이미 Fastify + Socket.IO를 바라보고 있으므로, 브리지는 백엔드 레이어 하나에서 끝내는 편이 프론트 수정 범위를 최소화함
  - 브리지 모드가 꺼져 있을 때는 기존 로컬 시뮬레이션이 그대로 남아 있어 개발 중 회귀 위험이 낮고, webhook은 후속 확장 포인트로 남길 수 있음

## [2026-03-10] 다크 모드는 별도 화면 분기 대신 전역 토큰 + `data-theme`로 적용
- 배경:
  - 설정에는 이미 `light/dark/system` 값이 있었지만, 실제 DOM과 CSS에는 연결되지 않아 화면이 항상 라이트 팔레트로 보였음
  - CEO 대시보드, 위젯, spatial office가 각각 다른 스타일 레이어를 갖고 있어 화면별 개별 분기를 늘리면 유지보수가 빠르게 무거워질 가능성이 있었음
- 결정:
  - `App`에서 저장된 `theme` 값을 읽어 `document.documentElement.dataset.theme`와 `color-scheme`에 반영
  - `system` 선택 시 `prefers-color-scheme: dark`를 구독해 OS 테마 변경도 즉시 따라가도록 구성
  - 다크 모드는 별도 CSS 파일을 만들지 않고 기존 디자인 토큰과 주요 패널에 `[data-theme="dark"]` override를 덧씌우는 방식으로 적용
- 이유:
  - 테마 해석 지점을 앱 루트 하나로 모아야 대시보드와 위젯이 같은 설정을 일관되게 따름
  - 토큰 기반 override는 이후 픽셀 스프라이트 교체나 3D 뷰 재설계 때도 색 체계만 재사용하기 쉬움
  - 화면별 분기보다 리스크가 낮고, 현재 구조에서 가장 작은 수정으로 전체 UI를 어둡게 전환할 수 있음

## [2026-03-10] 2D 오피스 자산은 CSS 박스 대신 16px SVG 픽셀 스프라이트로 관리

- 이유:
  - 현재 48px 타일을 16px 스프라이트의 3배 배율로 맞추면 흐림 없이 픽셀 감각을 유지할 수 있음
  - JSX 기반 스프라이트는 충돌/배치 데이터와 함께 수정하기 쉬워서 레이아웃 변경에도 대응이 빠름
  - 별도 PNG 세트를 늘리지 않아도 오피스 자산을 실제 픽셀 아트처럼 일관되게 관리할 수 있음

## [2026-03-01] ES 모듈 방식으로 Electron 메인 프로세스 구성
- 배경:
  - TypeScript 컴파일 후 Electron 실행 시 모듈을 찾을 수 없는 에러 발생
  - package.json에 `"type": "module"` 설정으로 ES 모듈 사용 중
- 결정:
  - import 경로에 `.js` 확장자 명시적으로 추가 (예: `./widget.js`)
  - ES 모듈에서 `__dirname` 사용을 위해 `fileURLToPath` + `dirname` 패턴 적용
- 이유:
  - ES 모듈 표준에서는 import 경로에 확장자가 필수
  - TypeScript는 `.ts` 파일을 컴파일할 때 import 경로를 변경하지 않음
  - `__dirname`은 CommonJS 전용이므로 ES 모듈에서는 `import.meta.url`로 대체 필요
