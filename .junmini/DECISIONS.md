# 결정 로그

## [2026-03-01] ES 모듈 방식으로 Electron 메인 프로세스 구성
- 배경:
  - TypeScript 컴파일 후 Electron 실행 시 모듈을 찾을 수 없는 에러 발생
  - package.json에 `"type": "module"` 설정으로 ES 모듈 사용 중
- 결정:
  - import 경로에 `.js` 확장자 명시적으로 추가 (예: `./widget.js`)
  - ES 모듈에서 `__dirname` 사용을 위해 `fileURLToPath` + `dirname` 패턴 적용
- 이유:
  - ES 모듈 표준에서는 import 경로에 확장자가 필수
  - TypeScript는 `.ts` 파일을 컴파일할 때 import 경로를 변경하지 않음
  - `__dirname`은 CommonJS 전용이므로 ES 모듈에서는 `import.meta.url`로 대체 필요

## [2026-03-01] Ollama로 로컬 LLM 연동
- 배경:
  - 사용자가 바탕화면에 카피바라 캐릭터를 넣고 로컬 LLM과 대화하고 싶어 함
  - 클라우드 API가 아닌 로컬에서 실행 가능한 솔루션 필요
- 결정:
  - Ollama를 사용해서 로컬 LLM(llama2) 연동
  - IPC를 통해 메인 프로세스에서 Ollama API 호출
  - 카피바라 SVG 아바타 컴포넌트 생성 및 상태별 애니메이션 추가
- 이유:
  - Ollama는 로컬에서 쉽게 LLM을 실행할 수 있는 오픈소스 도구
  - 렌더러에서 직접 HTTP 요청보다 IPC를 통한 호출이 보안상 안전
  - SVG로 카피바라를 그려서 애니메이션과 상태 변화를 쉽게 구현

## [2026-03-01] 3D 카피바라 캐릭터 구현 및 위젯 드래그 기능
- 배경:
  - 사용자가 위젯을 움직일 수 없다는 피드백
  - 2D SVG보다 3D 캐릭터가 더 생동감 있고 재미있음
- 결정:
  - Three.js 및 React Three Fiber 사용하여 3D 카피바라 모델 구현
  - `-webkit-app-region: drag` CSS 속성으로 위젯 드래그 기능 추가
  - 상태별 3D 애니메이션 구현 (호흡, 고개 흔들기, 점프, 회전)
  - 위젯 크기를 400x450으로 조정하여 3D 씬을 보기 좋게 표시
- 이유:
  - Three.js는 웹에서 3D를 구현하는 가장 인기 있는 라이브러리
  - React Three Fiber로 React 컴포넌트처럼 3D를 선언적으로 작성 가능
  - Electron의 frameless 창에서는 `-webkit-app-region`으로 드래그 영역 지정 필요
  - 3D 모델은 상태에 따른 다양한 애니메이션 표현이 용이함

## [2026-03-01] 다국어 지원 (i18n) 및 위젯 투명도 조절 기능 구현
- 배경:
  - 글로벌 사용자를 위한 영어 지원의 필요성이 대두됨
  - 바탕화면에서 항상 켜져 있는 위젯이 화면의 정보를 가리지 않도록 투명도 조절이 필요함
- 결정:
  - `useTranslation` 커스텀 훅과 `ko.json`, `en.json` 번역 파일을 활용한 경량 i18n 시스템 구축
  - 위젯에 `opacity` 스타일을 동적으로 적용하고, 설정 화면에 20~100% 범위의 슬라이더를 추가
- 이유:
  - 복잡한 i18n 라이브러리(i18next 등)를 쓰지 않고도 충분히 직관적이고 가벼운 구현이 가능함
  - 설정의 영속성을 보장하기 위해 기존 `settings.json`과 `useSettingsStore`에 상태를 통합하여 관리함

## [2026-03-01] 채팅 기록 로컬 저장 (Persistence)
- 배경:
  - 위젯이나 대시보드를 닫았다가 다시 열었을 때 이전 대화 내용이 사라짐
  - 사용자가 연속성 있는 대화 경험을 원함
- 결정:
  - Zustand의 `persist` 미들웨어를 사용하여 `localStorage`에 채팅 기록 저장
  - `useChatStore`를 생성하여 전역적으로 채팅 데이터 관리
- 이유:
  - 간단한 텍스트 데이터이므로 `localStorage`만으로도 충분히 빠르고 안정적임
  - Zustand를 사용하면 컴포넌트 간 동기화가 용이함

  - 역할은 기본값으로 제공하되 설정 화면에서 이름/역할/페르소나를 수정할 수 있게 유지
- 이유:
  - UI, 상태 시뮬레이션, 프롬프트가 서로 다른 동물 집합을 참조하면 사용자 경험과 모델 정합성이 깨짐
  - 종 특성과 역할을 프롬프트에 함께 넣어야 모델 문서에서 요구한 독특한 말투와 캐릭터성이 유지됨
  - 역할을 완전히 하드코딩하지 않고 수정 가능하게 두면 향후 기획 변경에도 대응하기 쉬움

## [2026-03-06] 6종 동물 기반 7개 에이전트 역할 체계 확정
- 배경:
  - 사용자 지시로 역할 구성이 `PM Agent`, `Research Agent`, `Trend Agent`, `Planning Agent`, `Document Agent`, `Coding Agent`, `Test Agent`로 구체화됨
  - 기존 구현은 6개 역할 기준이라 트렌드 전담 에이전트와 사용자 CEO 컨텍스트가 빠져 있었음
- 결정:
  - 카피바라는 `PM Agent`로 고정
  - 사용자는 시스템 안에서 사장(CEO)으로 인식
  - 6종 제한을 유지하기 위해 여우를 `Research Agent`와 `Trend Agent` 두 명으로 중복 사용
  - 나머지 역할은 호랑이=`Planning Agent`, 돼지=`Document Agent`, 고양이=`Coding Agent`, 개=`Test Agent`로 배치
- 이유:
  - 사용자가 명시한 역할 체계를 그대로 반영하는 것이 현재 기획 정합성에 가장 중요함
  - 동물 종을 추가하지 않으면서 7개 역할을 소화하려면 한 종의 중복 사용이 필요함
  - 사용자=사장 컨텍스트를 프롬프트와 UI에 넣어야 보고/지시 구조가 자연스러워짐
